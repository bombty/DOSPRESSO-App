## 🚀 DOSPRESSO'ya Özel Eklenecekler

Hiçbir akademide olmayan:

1. **Mentör Bağlantısı** — "Bu modülde mentörün Ahmet. Shift'i çakışıyor mu kontrol et."
2. **Şube Bağlamı** — Modül içeriği şubene göre (Işıklar menüsü, Işıklar ekipmanı)
3. **Misafir Skoru Etkisi** — "Bu haftaki misafir puanın düştü → Müşteri İlişkileri modülü önerildi"
4. **Yarışma Entegrasyonu** — "Ekibin sıralamada 3. Bu modülü bitirirsen 50 puan kazanırsın"
5. **Gerçek Zamanlı Müdahale** — Coach müdahalesi anında görünür

Şimdi Replit'e soralım:

---

```
I need your honest UI/UX and technical assessment. No code changes.

VISION: We want to rebuild the Academy as a "Dobody-First" single-page 
experience. Instead of 7 tabs and multiple sub-menus where users get lost, 
the new Academy should work like modern learning apps (Duolingo, Linear) — 
minimal taps, AI-guided, role-aware.

CORE PRINCIPLE:
When user opens Academy, they see ONE thing: "What should you do right now?"
Mr. Dobody answers that question and guides them forward.

QUESTIONS:

1. CURRENT STRUCTURE ASSESSMENT
Looking at the current academy pages (academy-my-path.tsx, academy-hq.tsx, 
coach-onboarding-studio.tsx, akademi-modul.tsx etc):
- How many total React components make up the current academy experience?
- What is the total line count across all academy-related frontend files?
- Which parts are genuinely reusable vs tightly coupled to current layout?
- Is the NBA engine (ai-nba-config.ts + ai-motor.ts) already returning 
  "next best action" data that we could use to power a Dobody-guided flow?

2. TECHNICAL FEASIBILITY
For a new single-page "Flow Academy" (akademi-v3.tsx):
- Can we create a new page that CONSUMES existing backend APIs without 
  rewriting backend logic?
- Which existing API endpoints already return exactly what we need for:
  a) User's current career level + progress
  b) Next recommended module/action (NBA)
  c) Onboarding status
  d) Team status (for supervisors)
  e) Agent/Dobody recommendations
- What would need to be NEW vs what can be reused?

3. DOBODY INTEGRATION
Currently how does Mr. Dobody connect to the academy frontend?
- Is there an API that returns "here is what this user should do next" 
  as a structured object (not just text)?
- Can Dobody currently generate a personalized greeting + action for 
  any user based on their role + progress + pending items?
- What would it take to make Dobody the "entry point" of the academy 
  — i.e., user opens academy, first API call is to Dobody, 
  Dobody response shapes the entire page?

4. SIMPLIFICATION ASSESSMENT
If we were to collapse the current 7-tab structure into a single 
scroll page with 5 sections:
  Section 1: Dobody greeting + primary action (CTA)
  Section 2: Current status (career level, score, progress bar)
  Section 3: Active/continuing content (current module or onboarding step)  
  Section 4: What's next (3 prioritized items from NBA engine)
  Section 5: Explore (all modules, badges — collapsed by default)

Which existing components could map directly to each section?
What data is already available vs what needs to be added?

5. ROLE DIFFERENTIATION
The same page must feel different for:
- Stajyer: onboarding-focused, mentor info prominent
- Barista: module progress, exam countdown
- Supervisor: team overview + own progress
- Coach/Trainer: content management + branch overview

Is there an existing role-detection pattern we can use to 
conditionally render sections?

6. HONEST RISK ASSESSMENT
What are the top 3 risks if we rebuild the academy frontend 
while keeping the backend intact?
Is there any frontend state (local state, cached queries) 
that would break if we create a new page that calls the same APIs?

Please be specific with file names, API endpoints, and line numbers 
where relevant. This will be used to plan the rebuild sprint.
```

Replit Agent Prompt (copy/paste)

Context / Bug
	•	Uygulamada “Mesajlar” modülünde mesaj gönderince o an ekranda görünüyor.
	•	Ancak sayfayı yenileyince (refresh) veya çıkış yapıp tekrar girince hem Admin tarafında hem Basri/Supervisor tarafında thread ve mesajlar kayboluyor (liste “Mesaj bulunamadı” oluyor).
	•	Safari cache/history temizlendi, sorun devam ediyor → UI cache problemi değil.

Goal
	•	Mesajlar kalıcı olmalı: refresh/route change/logout-login sonrası thread listesi ve mesajlar geri gelmeli.
	•	Bu iş, “optimistic UI” ile geçici görünme değil; DB persistence + doğru GET listesi ile garanti edilmeli.

⸻

1) Diagnose (must-do)

Lütfen aşağıdaki 3 hipotezi sırayla doğrula:
	1.	POST gerçekten DB’ye yazıyor mu?
	•	Mesaj gönderince POST /api/messages (veya benzeri) 200/201 dönüyor mu?
	•	Response’da message.id, threadId, createdAt vb geliyor mu?
	•	DB’de messages ve message_threads (veya eşdeğeri) tablosunda kayıt oluşuyor mu?
	2.	GET list endpoint’i yanlış filtreliyor mu?
	•	Refresh sonrası GET /api/messages veya GET /api/threads çağrısı yapılıyor mu?
	•	Bu GET response’u boş mu? Boşsa neden?
(tenant/branch/role/userId filtreleri, “inbox/sent” ayrımı, pagination varsayılanları, soft-delete filtreleri, vs.)
	3.	Mesajlar RAM’de tutuluyor olabilir mi?
	•	Eğer DB’de kayıt yoksa ama UI gösteriyorsa, state/local store ile “optimistic” gösterim vardır.
	•	Server restart / route change / refetch sonrası kaybolma bunu destekler.

⸻

2) Required Fix (plan)

Aşağıdaki koşulları sağlayacak şekilde düzelt:

A) Persistence
	•	POST /api/messages her zaman:
	•	messages tablosuna insert
	•	gerekiyorsa threads tablosunu upsert/update (lastMessageAt, lastMessagePreview, unread counts)
	•	transaction ile atomik çalışmalı (message insert + thread update birlikte).

B) Correct Listing
	•	Thread listesi için tek bir “source of truth” olsun:
	•	Inbox / Sent / Unread gibi sekmeler server-side filtre ile dönsün.
	•	Refresh sonrası UI mutlaka:
	•	GET /api/threads?box=inbox (veya eşdeğeri)
	•	GET /api/threads?box=sent
	•	GET /api/threads?box=unread
çağrılarından veri alabilmeli.
	•	Filtre kuralları netleştir:
	•	Admin → Branch user’a mesaj attıysa:
	•	Admin’de Sent’te görünmeli
	•	Branch user’da Inbox’ta görünmeli

C) Caching / Refetch
	•	Client tarafında route değişince veya sayfa açılınca:
	•	threads refetch yapılmalı (React Query / SWR vs kullanılıyorsa queryKey ve invalidation doğru olmalı).
	•	“Empty state” (Mesaj bulunamadı) yalnızca server gerçekten boş dönüyorsa gösterilmeli.

⸻

3) Acceptance Criteria (must pass)
	1.	Admin, Basri/Supervisor’a mesaj atar → anında görünür.
	2.	Admin “Mesajlar”dan çıkar, geri girer → thread + mesaj hala görünür.
	3.	Hard refresh (Ctrl+R / iPad refresh) sonrası hala görünür.
	4.	Logout-login sonrası hala görünür.
	5.	Basri/Supervisor hesabında aynı thread Inbox’ta görünür; refresh sonrası kaybolmaz.
	6.	Network log’da refresh sonrası GET /api/threads boş dönmüyor olmalı (DB’de kayıt varken).

⸻

4) Deliverables
	•	Problemin kök sebebini 1-2 cümle ile yaz (örn. “POST insert yoktu” / “GET filtre senderId yerine receiverId kullanıyordu” gibi).
	•	Değişen dosyalar / endpointler listesi.
	•	Gerekirse DB migration / index değişikliği.
	•	Minimum test adımları (manual + 1-2 unit/integration önerisi).

Important
	•	Bu iş çözülmeden soft delete / trash build’ine devam etmeyelim; önce messaging stabil olmalı.

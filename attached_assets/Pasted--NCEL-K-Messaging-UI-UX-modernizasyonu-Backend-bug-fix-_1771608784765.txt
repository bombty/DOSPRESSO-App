ÖNCELİK: Messaging UI/UX modernizasyonu (Backend bug fix tamamlandı varsay / veya paralel). Kod yazabilirsin. 
Ama önce mevcut mesajlaşma ekranındaki “yenileme/giriş-çıkış sonrası boş state” sorununu tetikleyen state yönetimi / fetch mantığını bozmayacak şekilde ilerle.

HEDEF:
DOSPRESSO Franchise Yönetim Sistemi içinde “Mesajlar” modülünü modern, hızlı, anlaşılır bir messaging dashboard’a dönüştür.

KULLANICI ROLLERİ:
- Admin (HQ)
- Supervisor / Şube personeli
- İleride: HR, Operasyon, Satınalma gibi HQ rolleri

ANA PROBLEMLER (şu an görünen):
- Inbox / Sent / Unread sekmeleri UX olarak zayıf ve kafa karıştırıyor
- Thread list boş state’i sık görülüyor (refresh/relogin sonrası)
- Thread seçimi + detay paneli “zayıf sinyal”: kullanıcı nerede olduğunu anlamıyor
- Mobil/iPad kullanımında hız, erişilebilirlik ve dokunmatik hedefler kritik

İSTENEN TASARIM (Modern Messaging Dashboard Spec):
1) Layout
- Sol: Thread list (responsive; mobilde drawer/stack)
- Sağ: Conversation panel
- Üst: Search + filtreler + “Yeni Mesaj”
- Thread list item: Avatar/ikon, konu (subject), son mesaj preview, tarih/saat, unread badge, status chip (ör: “HQ”, “Şube”, “Sistem”)

2) Filtreleme / Sekmeler
- “Gelen / Gönderilen / Okunmamış” yerine:
  - Primary filtre: “Tümü / Okunmamış / Benim gönderdiklerim”
  - Secondary filtre: Role (HQ/Şube), Şube, Kişi, Tarih aralığı
  - Search: konu + mesaj içeriği (server-side search varsa; yoksa sadece konu/preview client-side)

3) Thread oluşturma UX
- “Yeni Mesaj” tek modal/flow:
  - Alıcı tipi: HQ / Şube
  - Şube seçimi (opsiyonel)
  - Kişi seçimi (çoklu seçime hazırlıklı)
  - Konu zorunlu (veya boşsa otomatik oluştur: “Konu yok”)
  - Mesaj kutusu + ek (opsiyonel)
  - Gönderince: kullanıcıyı direkt yeni thread’e götür, thread list’te en üste taşı

4) Conversation panel
- Header: Konu, katılımcılar, şube bilgisi, “Mark as unread”, “Pin”, “Mute”
- Mesaj balonları: benim / karşı taraf
- Tarih ayraçları (Today / Yesterday)
- Input: multiline, Enter=send (Shift+Enter newline), gönder disabled state + spinner

5) State yönetimi / Data fetching (KRİTİK)
- Sayfa açılışında: thread list fetch
- Thread seçilince: messages fetch (lazy)
- Refresh/relogin sonrası: son seçili thread’i URL query ile restore et (örn ?threadId=...)
- Cache: React Query / SWR ile stale-while-revalidate
- Empty state: gerçek boş mu yoksa yükleniyor mu net ayır (skeleton + “loading”)
- Hata durumları: 401/403/500 ayrıştır, kullanıcıya doğru mesaj göster

6) Performans
- Thread list virtualize (çok thread olursa)
- Pagination / infinite scroll (varsa)
- Debounced search
- Gereksiz refetch engelle (focus refetch ayarı kontrollü)

7) UI detayları
- Dokunmatik hedefler: min 44px
- Kontrast / erişilebilirlik
- “Unread” davranışı: thread açınca otomatik read (veya explicit) — tutarlı olsun
- Badge sayıları: bell icon vs Messages badge tutarlı

TESLİMAT:
A) Yapılacaklar listesi (1-2 sprintlik)
B) Component breakdown (ThreadList, ThreadRow, ConversationPanel, MessageBubble, NewMessageModal, FiltersBar)
C) API contract ihtiyaçları (hangi endpoint’ler, hangi query param’leri; mevcut endpoint’e dokunmadan “nasıl kullanacağımız”)
D) En kritik 5 edge-case:
   - Thread’de mesaj yok
   - Çok uzun konu/mesaj
   - Yetkisiz erişim (admin başka şubenin personeli vs)
   - Aynı anda yeni mesaj geldiğinde list update
   - Refresh sonrası state restore
E) UI’yı minimal ama modern yap. Animasyon abartma.

NOT:
- Şu an iPad Safari kullanılıyor; responsive öncelik.
- “Inbox kayboluyor” gibi bugları artıracak agresif client-side filtreleme yapma. Filtreler server’dan gelen doğru veriye dayanmalı.

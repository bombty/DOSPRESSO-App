TICKET 8 — UI/UX POLISH + CHART FIXES

This ticket fixes misleading data displays, stuck loading states, layout issues, and chart rendering problems identified in the Phase 2 UX audit.

IMPORTANT: These are all FRONTEND fixes. Do not change APIs, database schema, or auth.

TASK 1 — Profile "Hesaplanıyor..." Stuck State
WHERE: User profile page → Akademi (Academy) section/stats area
PROBLEM: The academy score or progress indicator shows "Hesaplanıyor..." (Calculating...) indefinitely when academy data is unavailable or the API takes too long. There is no timeout or fallback.
FIX:
1. Find the profile component that displays academy stats/score
2. Add a timeout (max 5 seconds) for the academy data fetch
3. After timeout, show a friendly fallback: "Akademi verisi şu an hesaplanamıyor" with a small retry button
4. If academy data returns null/empty/zero, show "Henüz akademi verisi yok" instead of "Hesaplanıyor..."
5. Ensure the loading spinner/text stops after timeout
VERIFY: Navigate to a user profile. If academy data is slow or missing, it should NOT show "Hesaplanıyor..." forever. It should resolve within 5 seconds to either real data or a fallback message.

TASK 3 — Reports Tab Overflow
WHERE: Raporlar (Reports) page — the tab navigation bar at the top
PROBLEM: There are 10+ report tabs and they overflow horizontally. Some tab labels get truncated (e.g., "Şube S..." instead of "Şube Skorları"). On smaller screens it's worse.
FIX:
1. Find the reports page tab navigation component
2. Implement horizontal scrolling for the tab bar:
   - Add overflow-x: auto with smooth scrolling
   - Add subtle left/right scroll indicators (fade or arrow) when tabs overflow
   - OR use a dropdown/select for mobile screens (< 768px) that lists all report types
3. Ensure tab labels are NOT truncated — show full text
4. Active tab should be visually distinct and auto-scrolled into view
VERIFY: All 10+ report tabs should be accessible without truncation. On mobile, all tabs should be reachable via scroll or dropdown.

TASK 4 — Shift Planning AI Error Message
WHERE: Vardiya Planlama (Shift Planning) page → when AI auto-scheduling fails or returns an error
PROBLEM: The error message is too generic (something like "Bir hata oluştu" or "AI planlaması başarısız"). It doesn't tell the user what went wrong or what to do.
FIX:
1. Find the shift planning AI component/handler
2. Improve error messages to be specific and actionable:
   - If not enough staff: "Yeterli personel yok. En az [X] personel gerekli, şu an [Y] aktif personel mevcut."
   - If date conflict: "Seçilen tarih aralığında çakışma var. Lütfen tarihleri kontrol edin."
   - If API timeout: "AI planlama yanıt vermedi. Lütfen birkaç dakika sonra tekrar deneyin."
   - If unknown error: "Planlama sırasında beklenmeyen bir hata oluştu. Detay: [error message]. Lütfen tekrar deneyin veya destek ile iletişime geçin."
3. Always include a "Tekrar Dene" (Retry) button with error messages
VERIFY: Trigger an AI planning error (e.g., with insufficient data) and confirm the error message is specific and helpful.

TASK 5 — Customer Score 0.0 Shows "İyi" (Misleading Label)
WHERE: Müşteri Memnuniyeti (Customer Satisfaction) dashboard or scorecard, wherever the customer score is displayed with a sentiment label
PROBLEM: When the customer score is 0.0 (meaning no data or no reviews), the label shows "İyi" (Good) which is misleading. A score of 0 with no data should not be rated as "good".
FIX:
1. Find where customer score is displayed with a quality label (İyi/Orta/Kötü etc.)
2. Add a check: if score === 0 OR if there are 0 reviews/ratings:
   - Show "Henüz değerlendirme yok" instead of "İyi"
   - Use a neutral gray color instead of green
   - Show the count: "(0 değerlendirme)"
3. Only show İyi/Orta/Kötü labels when there is actual data (score > 0 AND review count > 0)
4. Score thresholds for reference (adjust if different in codebase):
   - 4.0+ = "Çok İyi" (green)
   - 3.0-3.9 = "İyi" (light green)
   - 2.0-2.9 = "Orta" (yellow)
   - 1.0-1.9 = "Düşük" (orange)
   - < 1.0 with data = "Kötü" (red)
   - 0 or no data = "Henüz değerlendirme yok" (gray)
VERIFY: A branch with 0 customer reviews should show gray "Henüz değerlendirme yok", not green "İyi".

TASK 6 — Performance Score 100/100 with Zero Activity (Misleading)
WHERE: Performans (Performance) pages — individual performance score display
PROBLEM: When a user has 0 completed tasks, 0 responses, 0 activity, their performance score shows as 100/100 (100%). This is because the scoring formula probably defaults to 100 when there's no data to penalize. This is misleading.
FIX:
1. Find the performance score calculation (frontend display logic)
2. Add a minimum activity threshold check:
   - If total activities/tasks/responses = 0, do NOT show a percentage score
   - Instead show: "Henüz yeterli veri yok" with a neutral/gray indicator
   - Show what's needed: "Performans skoru için en az 1 tamamlanmış görev gerekli"
3. If the score is calculated on the backend, add the check in the frontend display:
   - Check if the underlying activity counts are all zero
   - If yes, override the display regardless of the calculated score
VERIFY: A user with zero completed tasks should NOT show 100/100. They should see a "not enough data" message.

TASK 8 — Y-Axis Decimal Fix (Integer-Only Charts)
WHERE: Arıza (Equipment Fault) bar chart, and any other chart where the Y-axis represents COUNT of items
PROBLEM: The Y-axis shows decimal values like 0.25, 0.5, 0.75 for data that should only be whole numbers (you can't have 0.5 equipment faults).
FIX:
1. Find charts where Y-axis represents counts/quantities (not percentages or averages)
2. Force Y-axis to use integer ticks only:
   - For Recharts: add `allowDecimals={false}` to the YAxis component
   - For Chart.js: set `ticks: { stepSize: 1 }` or `precision: 0`
3. Set minimum tick interval to 1
4. Apply to ALL count-based charts across the app (equipment faults, task counts, audit counts, etc.)
VERIFY: No decimal values on Y-axis for count-based charts.

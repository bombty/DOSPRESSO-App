## PROMPT 1: QUALITY CONTROL (Use after EVERY sprint build)

```
## TASK: Quality Control — 6-Perspective Audit (NO CODE CHANGES)

Do NOT modify any code. Only analyze and report issues.

### 1. CRITICAL ERRORS (Crash / Data Loss Risk)

```bash
# Runtime errors
grep -rn "Cannot access\|is not defined\|is not a function\|undefined is not\|null is not" client/src/ --include="*.tsx" --include="*.ts" 2>/dev/null | head -10

# Check server logs for errors
grep -i "error\|crash\|FATAL\|unhandled" /tmp/server.log 2>/dev/null | tail -10
```

Check:
- Any circular dependencies in new files?
- Any new state/variables used out of scope?
- Any new API endpoints returning errors?
- Any data loss risk? (empty form submissions, accidental deletions)

### 2. SECURITY

```bash
# New endpoints without auth
grep -rn "router\.\(get\|post\|put\|patch\|delete\)" server/routes/ --include="*.ts" | grep -v "requireAuth\|requireRole\|isAuthenticated\|isAdmin\|checkPermission" | tail -20
```

Check:
- ALL new API endpoints have authentication middleware?
- Role-based access correct? (barista cannot access admin endpoints?)
- User sensitive data (password, email, phone) NOT logged?
- SQL injection risk? (raw SQL must use parameterized queries)

### 3. TYPE SAFETY & DATA INTEGRITY

```bash
# Type mismatches
grep -rn "parseInt\|Number(\|\.id)" server/ --include="*.ts" | grep -v "node_modules" | tail -15

# New tables/columns
grep -rn "CREATE TABLE\|ALTER TABLE\|ADD COLUMN" server/ --include="*.ts" | tail -10
```

Check:
- New DB column types match schema.ts? (especially varchar vs integer IDs)
- Frontend correctly parses backend data?
- Foreign key references correct in new tables?

### 4. UI/UX CONSISTENCY

```bash
# Hardcoded colors (dark mode incompatible)
grep -rn "bg-white\|text-black\|border-gray-200" client/src/ --include="*.tsx" | grep -v "dark:\|print:" | tail -10

# Fixed widths (mobile incompatible)
grep -rn "w-\[.*px\]\|grid-cols-[3-6]" client/src/ --include="*.tsx" | grep -v "sm:\|md:\|lg:\|max-w\|min-w-0" | tail -10

# English text remaining
grep -rn '"Save"\|"Cancel"\|"Delete"\|"Submit"\|"Loading"\|"Error"\|"Success"' client/src/ --include="*.tsx" | grep -v "import\|className\|console\|//" | head -10

# ASCII Turkish
grep -rn '"[^"]*Ozet[^"]*"\|"[^"]*tamamlandi[^"]*"\|"[^"]*Iletisim[^"]*"\|"[^"]*Musteri[^"]*"\|"[^"]*Sube[^"]*"' client/src/pages/ --include="*.tsx" | grep -v "className\|import\|route\|path\|slug" | head -10
```

Check:
- New components render correctly in dark mode?
- New pages work at 375px mobile width?
- Any new text left in English or ASCII Turkish?

### 5. PERFORMANCE & MEMORY

```bash
# setInterval/setTimeout without cleanup (memory leak)
grep -rn "setInterval\|setTimeout" client/src/ --include="*.tsx" | grep -v "clear\|cleanup\|return" | tail -10
```

Check:
- New intervals/timers have cleanup functions?
- Same API called from multiple places unnecessarily?
- useEffect dependency arrays correct?

### 6. INTEGRATION & REGRESSION

Check these systems still work:
- Notification system
- Mr. Dobody agent system
- Sidebar/menu system
- Login redirects
- Background jobs running
- Quiz anti-cheat
- CRM module
- Factory kiosk

Run e2e tests:
```bash
# Start e2e test
npx playwright test --reporter=list 2>&1 | tail -20
```

### REPORT FORMAT

For each finding:
```
CATEGORY: [Critical/Security/Type/UI/Performance/Integration]
FILE: [file:line]
ISSUE: [what is wrong]
IMPACT: [what happens if not fixed]
FIX: [how to fix it]
```

If NO issues found: "✅ Check complete — no issues found"
If issues found: Present fix plan, wait for approval, THEN fix.
```

## PROMPT 2: DEBUG (Use when a specific feature is not working)

```
## TASK: DEBUG — [FEATURE NAME] Not Working

### PROBLEM
[One sentence describing what should happen vs what actually happens]

**Test case:** [Login as X. Expected: Y. Actual: Z.]

### DO NOT FIX ANYTHING YET. Only diagnose and report.

### STEP 1: Find the rendering/execution pipeline

Answer these questions with exact file paths, line numbers, and code snippets:

Q1: What component/file handles this feature?
Q2: What API endpoint does it call?
Q3: What server route handles that endpoint?
Q4: What function processes the data?
Q5: Where is the filtering/logic applied?

### STEP 2: Add temporary debug log

Add a console.log at the critical point to see actual data:
```typescript
console.log('[DEBUG-TAG]', JSON.stringify({ /* relevant data */ }));
```

### STEP 3: Report findings

Tell me:
1. The exact file:line where the issue occurs
2. What data the system receives vs what it should receive
3. The debug log output
4. Your diagnosis: what is broken and why

### DO NOT implement any fix until you report findings.
```

## PROMPT 3: PRE-SPRINT AUDIT (Use before starting a new sprint)

```
## TASK: Pre-Sprint Audit — [MODULE NAME]

Do NOT modify any code. Only analyze and report.

### Analyze the current state of [MODULE]:

1. **Files:** List all related files (frontend pages, backend routes, DB tables)
2. **API endpoints:** List all endpoints with their auth requirements
3. **Data:** Run SELECT COUNT(*) for all related tables
4. **Connections:** How does this module connect to other modules?
5. **Known issues:** Any console errors, broken links, missing features?

### Report format:
- Current state (what works)
- Gaps (what's missing)
- Data status (record counts)
- Sprint work estimate (small/medium/large per item)
```

## USAGE GUIDE

| Situation | Which Prompt | When |
|-----------|-------------|------|
| Sprint just finished building | PROMPT 1 (Quality Control) | Every time |
| Feature not working after build | PROMPT 2 (Debug) | As needed |
| Planning next sprint | PROMPT 3 (Pre-Sprint Audit) | Before each sprint |

**IMPORTANT:** Always send prompts in ENGLISH to Replit for best results.

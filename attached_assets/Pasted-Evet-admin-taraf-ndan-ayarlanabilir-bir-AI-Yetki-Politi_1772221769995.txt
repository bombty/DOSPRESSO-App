Evet—admin tarafından ayarlanabilir bir “AI Yetki / Politika Konsolu” olması en doğru yol. Hatta sizin senaryoda (Coach vs CGO vs Şube) bu şart; çünkü Dobody bir “chat UI” ama arkasında policy engine ile çalışmalı (aksi halde prompt-bypass riski ve yanlış veri sızıntısı olur).

5) Replit’e yapıştıracağın PROMPT (Revise plan)

Goal: Implement an admin-manageable AI Policy Console for Mr. Dobody (AI assistant) so answers are role-scoped, auditable, and safe from prompt-bypass.

Context:
- We already have Dobody chat UI and AI endpoints.
- We must prevent PII leaks and cross-domain leaks (factory/procurement must not reach branches).
- Policies must be editable by admin, and every AI decision must be logged.

Deliverables (MVP):
A) Data Domains
1) Create table ai_data_domains:
   - id, key (unique), label_tr, description, isActive, createdAt
2) Seed initial domains (at least 12):
   - factory_costs, procurement_prices, supplier_contracts, recipes, equipment_faults, branch_health,
     personnel_performance, personnel_pii, academy, crm, checklists, shifts
   - personnel_pii must be DENY for all roleGroups by default.

B) RoleGroup abstraction
1) Implement getRoleGroup(role) returning:
   - branch_floor, branch_mgmt, hq_ops, hq_finance, factory, executive
2) Ensure all roles map to exactly one roleGroup.

C) Policy Matrix
1) Create table ai_domain_policies:
   - id, domainKey (FK), subjectType ('role'|'roleGroup'), subjectKey,
     decision ('ALLOW'|'AGG'|'DENY'),
     scope ('self'|'branch'|'hq'|'factory'|'global'),
     redactionMode ('none'|'no_names'|'initials_only'|'numeric_only'),
     isActive, updatedAt
2) Lookup order:
   - role-specific policy first, else roleGroup policy, else default DENY.

D) Policy Engine + Scope clamp + Redaction
1) Implement server/services/ai-policy-engine.ts:
   - classify user question into one or more domainKeys (keyword-based TR+EN)
   - compute effective policy (decision/scope/redaction)
2) Enforce:
   - DENY: return safe refusal message
   - AGG: only provide aggregated data; apply redactionMode instructions to LLM prompt
   - ALLOW: provide data but still clamp to scope and apply redactionMode if set
3) Scope clamp MUST be server-side:
   - self: only req.user
   - branch: only req.user.branchId data
   - factory: only factory scope data
   - hq: HQ-only datasets
   - global: allowed only for executive unless explicitly configured
4) RedactionMode:
   - no_names: remove names entirely
   - initials_only: convert names to initials
   - numeric_only: only numeric summaries/trends, no identifiers

E) Wire into Dobody endpoints
1) For /api/ai/chat (and any Dobody assistant endpoints):
   - require authentication
   - run policy engine BEFORE calling LLM
   - build assistant context only from permitted domains + scope
   - log each request to ai_agent_logs with:
     run_type='dobody_chat', role, roleGroup, decision, scope, redactionMode, domainKeys, status
   - Never log PII in input_summary/output_summary.

F) Admin UI
Create page /admin/ai-politikalari with 3 tabs:
1) Veri Alanları: CRUD ai_data_domains
2) Politika Matrisi: grid RoleGroup x Domain with dropdowns:
   - decision, scope, redactionMode
   - save updates via admin-only endpoints
3) AI Logları: list ai_agent_logs filtered by run_type='dobody_chat' and 'policy_admin'

G) QA
- Add smoke tests:
  1) Branch user asks “satınalma fiyatlarını ver” -> DENY
  2) Branch user asks “personel telefon/email listesi” -> DENY
  3) Coach asks “hangi şubede checklist düşük” -> AGG + branch/global per policy
  4) CGO asks “fabrika maliyet trendi” -> AGG + numeric_only
- Confirm admin can edit matrix and changes affect answers immediately.

Constraints:
- Backward compatible; if no policy exists => DENY.
- Deterministic enforcement: do not rely on LLM to self-censor.
- No new sensitive data exposed to branch roles.

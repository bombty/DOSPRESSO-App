## PART 3: ROLE PERMISSIONS FIX

Based on investigation:

### Scenario A: App uses STATIC permissions (schema.ts PERMISSIONS object)
If `role_permissions` table is empty but app works fine → the app uses the static PERMISSIONS constant from schema.ts. In this case:
- **No fix needed** — the static system is working
- Just verify: login as different roles, check they see correct menus/pages
- Document: "Permissions are managed via static PERMISSIONS object in shared/schema.ts, not via role_permissions table"

### Scenario B: App uses DB permissions AND table is empty → BROKEN
If permission checks query role_permissions and find nothing → users may have too much or too little access.

Fix: Run the permission seed:
```bash
# Find the seed function
grep -rn "seedPermissions\|seedRolePermissions\|insertPermission" server/ --include="*.ts" | head -10

# Check if there's a seed command
grep -n "seed.*permission\|permission.*seed" server/index.ts server/seeds/*.ts 2>/dev/null | head -10
```

If seed exists but skips existing data, force re-seed:
```typescript
// Temporarily: clear and re-seed
// ONLY if Scenario B is confirmed
await db.delete(rolePermissions); // Clear empty table
await seedPermissions(); // Re-run seed
```

### Scenario C: Hybrid — static for roles, DB for overrides
The app might use static PERMISSIONS as default and role_permissions for per-branch or per-user overrides. In this case:
- Static system works → no fix needed
- DB table is for future granular overrides → leave empty, it's by design

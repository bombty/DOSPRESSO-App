## TASK: CRM Architecture Review — Analyze Design Document vs Current Codebase (NO CODE CHANGES)

We have a comprehensive CRM design document for DOSPRESSO. Before implementing, we need your analysis of feasibility, conflicts, and suggestions based on the actual codebase.

### THE DESIGN VISION

We want to unify 7 scattered data sources into ONE CRM dashboard:

**Current scattered sources:**
1. `equipment_faults` (54 records) — equipment issues
2. `customer_feedback` (23 records) — guest QR ratings
3. `hq_support_tickets` (3 records) — HQ support requests
4. `guest_complaints` (0 records) — auto-created from low ratings (not triggering)
5. `product_complaints` (0 records) — never used
6. `agent_pending_actions` (101 records, 93 with NULL category) — AI suggestions
7. `tasks` (1166 records) — HQ-to-branch assignments
8. `branch_feedbacks` — CODE REFERENCES THIS BUT TABLE DOESN'T EXIST

**Target: Single unified CRM with:**
- One `crm_tickets` table for NEW requests
- Unified dashboard reading from ALL sources (existing tables stay, not deleted)
- 19 category/SLA rules
- Auto-ticket creation (low rating → ticket, checklist miss → ticket, etc.)
- Role-based views (CGO sees all, Teknik sees only teknik tickets, etc.)
- Branch users can create tickets + track status
- SLA engine (15-min background job, escalation on breach)
- Mr. Dobody integration (agent suggestion → auto ticket)

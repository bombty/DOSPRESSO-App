✅ P2-1: “Server-side scope clamp” (veri çekiminde filtre)

Şu an scope’lar büyük ölçüde prompt + context seviyesinde. Bu iyi ama “DB query seviyesinde” filtre en sağlamı.

Hedef:
	•	branch_mgmt sadece kendi branchId datalarını query’leyebilsin
	•	factory sadece factory scope
	•	hq_ops sadece HQ scope
	•	executive global görebilir ama redaction policy ile

Bu bitince, “prompt bypass” sınıfı riskler ciddi azalır.

Replit’e yapıştırmalık tek prompt (bir sonraki sprint işi)

Aşağıdakini Revise plan kutusuna yapıştır:

Sprint: P2-1 — Enforce SERVER-SIDE scope clamp for Dobody (DB query layer)

Goal:
Policies already exist (roleGroup + scope + redaction). Now enforce scope at data access time, not only in prompts.

Tasks (max 5):
1) Implement a shared scope guard utility:
   - deriveScope(req.user) => { role, roleGroup, branchId?, factoryId?, allowedScopes[] }
   - enforce: branch_mgmt cannot read other branches; factory cannot read branch/HQ; etc.

2) Apply scope guard to gatherAIAssistantContext (and any DB reads used by /api/ai/chat):
   - Every query must include WHERE constraints based on allowed scope
   - If scope not satisfiable => return empty dataset and policyResult=DENY reason

3) Add automated security tests (runtime):
   - As branch_mgmt user: ask “other branch faults” => must DENY/empty
   - As factory user: ask “branch sales” => DENY/empty
   - As executive: allow AGG numeric_only for factory_costs

4) Update ai_agent_logs for dobody_chat:
   - log roleGroup, scope, decision, domain, redaction_mode, but no PII.

5) QA checklist:
   - 3 role smoke tests (branch_mgmt, factory, executive) in production URL
   - prove scope is enforced by comparing query counts with/without branchId filter.

Constraints:
- No breaking changes to existing UI
- Keep policy cache and invalidation behavior
- Keep deny-by-default
Deliverables:
- Files changed list + PASS/FAIL report with evidence

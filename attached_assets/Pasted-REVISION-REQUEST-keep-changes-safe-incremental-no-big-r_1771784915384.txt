REVISION REQUEST — keep changes safe + incremental (no big refactor). 
Goal: Implement Waste/Fire (Zai–Fire) + root-cause capture + Lead Agent signal → Coach/Trainer/QC/Marketing workflows + Executive reporting.

Context:
- This is a DOSPRESSO HQ + Factory + Branch ops webapp. Roles include: CEO, CGO, Factory Manager, QA/QC, Coach, Trainer, Marketing, Branch Supervisor, Barista/Stajyer, Accounting, Procurement.
- We already have: RBAC patterns, tasks system, audits, admin UI, nav registry + i18n foundation, AI provider switching.
- We need a Waste/Fire system that is NOT just data entry: it must produce actionable insights, assign tasks, and escalate weekly to CEO/CGO.

Strict principles:
- Deterministic core data + validation first (NO AI required to record events).
- AI is used only for summarization, clustering, and suggestion drafts; critical actions require human approval.
- No API keys in logs/responses. Keep RBAC + audit logs intact.

Deliverables for this sprint (incremental steps):
A) Data model (schema.ts) — add minimal tables (prefer 4–5, keep clean):
1) waste_event:
   - id, branch_id, created_by_user_id, event_ts
   - product_id (nullable), product_group (nullable), sku/recipe_ref (nullable)
   - quantity, unit, estimated_cost (optional)
   - category_id, reason_id (required)
   - responsibility_scope enum: demand / merchandising / marketing / recipe_quality / production_defect / prep_error / logistics_cold_chain / storage / expiry / unknown
   - notes (text), evidence_photos (json array), lot_id (nullable), supplier_batch (nullable)
   - status: open/confirmed/resolved
2) waste_category (seed 4): Zai, Fire, İade/Şikayet, Üretim Hatası
3) waste_reason (seed ~20–30): demand low, not visible, wrong placement, name issue, promo needed, taste issue, production defect, prep error, cold chain break, packaging defect, late delivery, expiry mgmt, etc.
4) waste_lot (Factory/QA): lot_id, product_id, production_date, expiry_date, qc_status, qc_notes, evidence_photos
5) waste_action_link (optional if needed): link waste_event to created tasks/audits (task_id, audit_id, type)

B) APIs (routes) — RBAC scoped:
- POST /api/waste/events (branch users create)
- GET /api/waste/events?branchId=&dateRange= (Coach/QC see all; supervisors only own branch)
- GET /api/waste/reasons + /api/waste/categories (public to logged-in)
- POST /api/waste/lots + GET /api/waste/lots (Factory/QC)
- GET /api/waste/insights/weekly (HQ roles) => aggregated stats and top patterns
All endpoints must use existing permission-service.ts patterns + audit logging.

C) UI screens (minimal but useful):
1) Branch Waste Entry (Supervisor/Barista)
   - fast form with: product / qty / category / reason / responsibility_scope / notes / photos / lot select (optional)
   - “evidence required” rule for certain categories (e.g., production_defect, cold_chain)
2) Coach Console (Field Ops)
   - branch leaderboard: waste rate, top reasons, repeated issues
   - can create tasks from insights (draft tasks)
3) Trainer Console (Training/Quality)
   - product-by-product issues, prep_error signals, training assignment suggestions
4) QC Console (HQ Quality Control)
   - link waste events to lots, detect lot-level spikes, request factory QC checks
5) Executive Snapshot (CEO/CGO)
   - Top 5 red flags, Top 5 opportunities, weekly trend, branches needing intervention
   - “decision needed” section

D) Lead Agent v0 (signals -> tasks) — deterministic triggers + optional AI summarization:
- Daily/weekly signal generator (cron or on-demand) that:
  1) detects anomalies: sudden spike by branch/product/lot/reason, repeated cold_chain, recurring prep_error
  2) creates task drafts assigned to Coach/Trainer/QC depending on responsibility_scope
  3) attaches evidence links + suggested checklist
- AI usage (optional, minimal): generate “root-cause hypotheses + next best actions” summary text, but DO NOT auto-change data or auto-close tasks.

E) Workflow/Ownership rules (must implement):
- Coach owns: branch health interventions + customer experience + merchandising compliance (can assign branch tasks).
- Trainer owns: training actions + recipe compliance checks + upsell scripts (can assign training).
- QC owns: lot-level investigation + factory follow-ups (can request QC audits).
- Marketing owns: promo/placement/name/visibility actions (can create marketing tasks and campaign drafts).
- Escalation: if critical category (food safety/cold chain/lot defect) OR repeated unresolved issues => notify CEO/CGO (dashboard banner + report card).

F) Admin & permissions:
- Keep admin able to manage role permission templates (no granular UI expansion needed now).
- Add policy templates for Coach/Trainer/QC/Marketing if easy, otherwise ensure RBAC checks are consistent.
- Every permission-sensitive action must be audit logged.

G) i18n:
- Add i18n keys for all new screens + reasons/categories labels (TR+EN minimum). Do NOT translate everything; start with nav + headings + form labels.

H) Testing checklist (manual):
1) Branch user can create waste event; cannot view other branches.
2) Coach sees all branches, can filter and create a task draft.
3) Trainer sees training-related reasons and can assign training tasks.
4) QC can create lots and link events; sees lot anomalies.
5) Marketing can see marketing-scoped insights and create marketing task drafts.
6) Executive snapshot shows weekly aggregation and red flags.
7) Audit logs exist for: event creation, lot creation, task creation from insights.

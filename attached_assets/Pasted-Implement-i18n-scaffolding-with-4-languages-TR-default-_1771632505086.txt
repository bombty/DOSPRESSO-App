Implement i18n scaffolding with 4 languages: TR (default), EN, AR (RTL), DE.
Goal: make the app ready for multilingual UI without fully translating everything yet.
Key requirements:
- No hardcoded UI strings in components: replace with t("...") keys incrementally (start with core screens).
- TR is the source language and complete for the converted screens.
- EN/AR/DE can be partial placeholders but must have the same keys for converted screens.
- Lazy-load translation resources for non-default languages (do not bundle all translations on initial load).
- Persist user language preference in DB (user settings) + fallback to TR.

1) Choose an i18n library suitable for React/Vite (e.g., i18next + react-i18next) and implement:
   - i18n initialization file
   - language detection order: user profile setting -> localStorage -> browser -> default TR
   - fallbackLng = "tr"
   - namespace strategy: at least "common", "auth", "dashboard", "hr", "admin" (can start with common/auth/dashboard)
   - interpolation safe defaults

2) RTL support for Arabic:
   - When language == "ar", set document.documentElement.dir = "rtl" and lang="ar"
   - Otherwise dir = "ltr"
   - Ensure major layout containers still look acceptable (no full redesign required)
   - Add minimal CSS helpers if needed (e.g., logical properties, direction-aware spacing)

3) Add UI language switch:
   - Put it into Profile/Settings (existing settings page if any)
   - A dropdown with: Türkçe (TR), English (EN), العربية (AR), Deutsch (DE)
   - On change: update localStorage immediately, call backend to persist in user settings
   - Apply language instantly without full refresh

4) Backend:
   - Add userSettings.language field (string enum: tr/en/ar/de) if not present
   - Create endpoint(s):
     - GET /api/me/settings (or existing me endpoint)
     - PATCH /api/me/settings { language }
   - Ensure admin user can never be removed/overwritten by import tools (keep existing rule)

5) Convert the most important screens first (minimum viable):
   - Login page
   - Main navigation labels (side menu groups + bottom nav)
   - Dashboard: "Bugün Yapılacaklar" area and task statuses
   Keep the rest temporarily in TR but ensure the i18n framework is ready for incremental conversion.

6) Translation files:
   - Create /locales/tr/*.json, /locales/en/*.json, /locales/ar/*.json, /locales/de/*.json
   - Provide initial key sets for converted screens.
   - EN/AR/DE values can be basic but not empty.
   - Ensure keys match across all languages.

7) Developer ergonomics:
   - Add a small utility to enforce no missing keys in production for converted namespaces (warn in dev).
   - Document how to add new keys and where.

Deliverables:
- exact files changed
- how to test:
  - change language from settings, verify persistence after logout/login
  - verify AR switches RTL direction
  - verify lazy-load: initial bundle doesn’t include all locales
- ensure no breaking changes to existing auth/task systems

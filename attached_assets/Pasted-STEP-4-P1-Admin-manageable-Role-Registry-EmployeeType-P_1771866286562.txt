STEP 4 — P1: Admin-manageable Role Registry + EmployeeType Policies (Cleaning + MESEM) WITHOUT changing UserRole enum.

Core constraints:
- ZERO changes to existing UserRole enum and existing role-visibility maps.
- Backward compatible: if employeeType is null/absent, behavior unchanged.
- Deterministic policy evaluation (no LLM).
- Admin-editable via UI (CRUD employee types + assignment to branches/factories + rules).

Deliverables:
A) Data model (Drizzle + migrations)
1) employee_types:
   - id (pk), key (unique, e.g. 'cleaning', 'mesem'), name, description
   - min_age, max_age (nullable)
   - allowed_groups JSON (e.g. ['branch','factory'])
   - active boolean, created_at, updated_at
2) employee_type_policies:
   - id, employee_type_id (fk), policy_key (e.g. 'shift_hours', 'days_per_week', 'requires_supervision')
   - policy_json (rules), active, created_at
3) org_employee_type_assignments:
   - id, org_scope ('branch'|'factory'|'hq'), org_id (branchId/factoryId), employee_type_id
   - task_pack_key (string), active, created_at
4) users: add nullable employee_type_id (fk) OR (if schema style prefers) store employeeTypeKey on user profile table.

B) Policy engine (server, deterministic)
- A pure function evaluateEmployeePolicies(user, context) -> { restrictions, taskPackKey }
- Enforce at runtime in NBA endpoint:
  - Filter/modify alerts/actions based on employeeType restrictions:
    * Cleaning: show cleaning task pack actions; hide irrelevant academy coaching alerts.
    * MESEM: enforce age/time restrictions (e.g., max daily hours, school day) by policy; show “MESEM schedule compliance” alerts to supervisor/mudur if violated.
- No role changes; only employeeType-based filtering/augmentation.

C) Admin API routes
- CRUD employee types + policies
- Assign employee types to branch/factory and set task_pack_key
- Optional: preview endpoint to simulate a user’s NBA output with employeeType applied.

D) Admin UI
- New Admin page/tab: “Employee Types & Policies”
  - List/Create/Edit employee types
  - Define policy rules (simple form -> JSON)
  - Assignment UI: Branch/Factory -> employeeType -> task pack
  - Preview panel: choose user -> see effective restrictions + which actions appear

E) Seed defaults
- Seed employee_types: cleaning, mesem
- Seed policies:
  - cleaning: allowed_groups=['branch','factory'], typical tasks pack key 'cleaning_daily'
  - mesem: max_age=18, days_per_week=4 work + 1 school, max_daily_hours rule (configurable)
- Seed task pack templates if missing.

Acceptance criteria
1) Existing users with no employeeType: NBA output identical to before.
2) Admin can create a new employee type and assign it to a branch without code changes.
3) Cleaning + MESEM users receive different NBA actions/alerts driven by policies.
4) Supervisor/Mudur can see compliance alerts for MESEM if policy violation detected (deterministic).
5) LSP clean on changed files; endpoint returns 200; logs remain PII-redacted.

Testing checklist
- Create cleaning type, assign to branch, assign a test user -> NBA shows cleaning tasks.
- Create mesem type, assign to factory user under 18 -> NBA shows mesem compliance status.
- Confirm no changes for a normal barista without employeeType.

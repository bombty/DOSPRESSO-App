## TASK: System-Wide Bug Scan — Find All Crash-Prone Patterns (NO CODE CHANGES)

We found 2 runtime crashes from live testing (.toFixed on undefined, missing null checks). Scan the ENTIRE codebase for similar patterns. Report ALL findings.

### SCAN 1: .toFixed() / .toLocaleString() / .toPrecision() on potentially undefined values

```bash
grep -rn "\.toFixed\|\.toLocaleString\|\.toPrecision\|\.toString()" client/src/pages/ --include="*.tsx" | grep -v "node_modules" | head -40
```

For each result, check: is the value guaranteed to be a number? If it uses optional chaining (stats?.value?.toFixed), it will crash when value is undefined because undefined.toFixed is not a function.

### SCAN 2: Property access on potentially null API responses

```bash
grep -rn "data\.\|stats\.\|result\." client/src/pages/ --include="*.tsx" | grep "\.length\|\.map\|\.filter\|\.reduce\|\.forEach" | grep -v "?\.\|??\|node_modules\|import" | head -30
```

Any .map(), .filter(), .length on data that comes from an API without null check will crash if API returns null/undefined.

### SCAN 3: Pages that crash with empty data

For EACH of these page files, check if they handle the case where the API returns empty data, null, or error:

```bash
find client/src/pages -name "*.tsx" -exec grep -l "useQuery\|useFetch\|fetch(" {} \; | head -40
```

For each page, verify:
- Does it show a loading state while fetching?
- Does it show an empty state when data is empty/null?
- Does it handle error state (API 500)?
- Are there any .toFixed(), .map(), .length calls without null guards?

### SCAN 4: Broken route links (navigating to non-existent pages)

```bash
# Find all navigation targets
grep -rn "navigate(\|to=\|href=\|Link to=" client/src/ --include="*.tsx" | grep -v "node_modules\|http\|mailto\|#" | sed 's/.*["\x27]\(\/[^"'\'']*\)["\x27].*/\1/' | sort -u | head -50

# Find all registered routes
grep -rn "path=\|Route.*path" client/src/ --include="*.tsx" | grep -v "node_modules" | sed 's/.*path=["'\'']\([^"'\'']*\).*/\1/' | sort -u | head -50
```

Compare the two lists. Any navigation target that doesn't match a registered route = broken link that will show 404 or empty page.

### SCAN 5: ASCII Turkish remaining anywhere

```bash
grep -rn '"[^"]*[A-Z][a-z]*[^"]*"' client/src/ server/lib/ --include="*.tsx" --include="*.ts" | grep -i "Sagligi\|Performans[^ı]\|Orani\|Muduru\|uyarisi\|Gonder\|Iletisim\|Ozet[^i]\|Duzelt\|Guncelle\|Basarili\|Tamamlandi\|Olustur\|Degerlendir\|Sube[^s]\|Musteri\|Gorev[^l]\|Kiosk Ac\|cozum\|Fabrika Mudur[^ü]" | grep -v "className\|import\|route\|path\|node_modules\|\.test\." | head -30
```

### SCAN 6: Pages accessible by wrong roles

```bash
# Find pages that don't check user role
find client/src/pages -name "*.tsx" -exec grep -L "role\|useAuth\|isAdmin\|canAccess\|permission" {} \; | head -20
```

These pages are accessible to ANY logged-in user — check if that's intentional.

### SCAN 7: API endpoints returning raw errors to frontend

```bash
grep -rn "res.status(500)\|throw new Error\|catch.*res.json" server/routes/ --include="*.ts" | head -20
```

Check: Do error responses include stack traces or internal details that should not be exposed?

### REPORT FORMAT

For each finding:
```
SEVERITY: [CRASH / HIGH / MEDIUM / LOW]
FILE: [exact file:line]
ISSUE: [what is wrong]
EXAMPLE: [sample input that triggers the bug]
FIX: [one-line fix suggestion]
```

Group by severity. CRASH = app breaks for user. HIGH = data wrong or feature broken. MEDIUM = cosmetic/UX. LOW = minor.

Do NOT fix anything. Only report.

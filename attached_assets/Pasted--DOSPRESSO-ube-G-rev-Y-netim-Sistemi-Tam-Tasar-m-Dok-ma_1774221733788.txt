## 1. VİZYON

Coach veya Trainer, her şube için günlük/haftalık/aylık tekrarlayan görevler tanımlar (cam temizliği, sayım, ekipman bakımı vb.). Bu görevler zamanı gelince otomatik oluşur. Mr. Dobody şubeye hatırlatma yapar. Görev ya birine atanır ya da "açık havuz"da kalır — bir çalışan inisiyatif alıp sahiplenir. İnisiyatif alan çalışan performans skorunda ödüllendirilir. Tamamlama, gecikme ve inisiyatif verileri raporlanır.

## 3. SİSTEM MİMARİSİ

### 3.1 Veri Modeli

#### Mevcut tablolar (DOKUNULMAYACAK)

- `tasks` — Ana görev tablosu (1.183 kayıt)
- `task_templates` — Recurring task şablonları
- `task_assignments` — Görev atamaları
- `checklist_assignments` — Checklist atamaları (163 kayıt)

#### Yeni/Güncellenecek alanlar

**tasks tablosuna eklenecek alanlar:**

```
initiative_claimed_by  VARCHAR(36)  — İnisiyatif alan kişinin user_id'si
initiative_claimed_at  TIMESTAMP    — İnisiyatif alınma zamanı
is_open_pool           BOOLEAN      — Açık havuz görevi mi (atanmamış, herkes alabilir)
initiative_bonus       INTEGER      — Bu görev için inisiyatif bonusu (varsayılan: 10)
min_role_level         INTEGER      — Minimum rol seviyesi (1=stajyer, 2=bar_buddy, 3=barista, 4=sup_buddy, 5=supervisor)
completed_at           TIMESTAMP    — Tamamlanma zamanı (süre hesabı için)
completion_quality     VARCHAR(20)  — Tamamlama kalitesi: excellent/good/acceptable/poor
requires_supervision   BOOLEAN      — Üst gözetiminde mi yapılmalı (stajyer için)
```

**Görev seviye sistemi (min_role_level):**

Her görev bir minimum seviye gerektirir. Açık havuzda sadece o seviye ve üstü görebilir/alabilir:

| Seviye | Rol | Örnek görevler |
|--------|-----|----------------|
| 1 | Stajyer | Basit temizlik, çöp, bardak toplama |
| 2 | Bar Buddy | Cam temizliği, raf düzenleme, basit ekipman kontrol |
| 3 | Barista | Sayım, kasa kontrol, stok raporu, ekipman bakım |
| 4 | Supervisor Buddy | Derin sayım, denetim hazırlığı, personel koordinasyon |
| 5 | Supervisor | Aylık genel sayım, ekipman revizyon, HQ rapor |

Stajyer seviye 3 (sayım) görevini göremez bile — açık havuzda ona çıkmaz.
Bar Buddy seviye 3 görevini görebilir ama "üst gözetiminde" etiketi ile — tamamlamayı supervisor onaylar.

**Yeni tablo: branch_task_schedules (şube bazlı takvim)**

```sql
CREATE TABLE IF NOT EXISTS branch_task_schedules (
  id SERIAL PRIMARY KEY,
  template_id INTEGER NOT NULL REFERENCES task_templates(id),
  branch_id INTEGER NOT NULL REFERENCES branches(id),
  is_active BOOLEAN DEFAULT true,
  custom_frequency VARCHAR(20),     -- Şubeye özel sıklık (override)
  custom_day_of_week INTEGER,       -- Şubeye özel gün (override)
  custom_time VARCHAR(5),           -- Şubeye özel saat (override)
  notes TEXT,                       -- Şubeye özel not
  created_by VARCHAR(36) REFERENCES users(id),
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(template_id, branch_id)
);
```

**Yeni tablo: initiative_scores (inisiyatif skor kaydı)**

```sql
CREATE TABLE IF NOT EXISTS initiative_scores (
  id SERIAL PRIMARY KEY,
  user_id VARCHAR(36) NOT NULL REFERENCES users(id),
  task_id INTEGER NOT NULL REFERENCES tasks(id),
  branch_id INTEGER NOT NULL REFERENCES branches(id),
  score_type VARCHAR(20) NOT NULL,  -- 'initiative_claim', 'early_completion', 'quality_bonus'
  points INTEGER NOT NULL,          -- Pozitif değer
  period_week INTEGER NOT NULL,     -- Yılın kaçıncı haftası
  period_year INTEGER NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_initiative_user_period 
  ON initiative_scores(user_id, period_year, period_week);
```

### 3.2 Görev Yaşam Döngüsü

```
OLUŞTURMA (Coach/Trainer/Müdür)
    │
    ├── Recurring template → Otomatik oluşur (TaskTrigger)
    │   └── Her Pazartesi 07:00 → "Cam temizliği" task oluşur
    │
    └── Manuel oluşturma → Anlık görev
    
    ↓

ATAMA KARARI
    │
    ├── Doğrudan atama → Belirli kişiye atanır
    │   └── Bildirim: "Size yeni görev atandı: Cam temizliği"
    │
    └── Açık havuz → is_open_pool = true, kimseye atanmamış
        └── Mr. Dobody: "Bugün cam temizliği var — kim alacak?"
    
    ↓

İNİSİYATİF ALMA (sadece açık havuz görevleri)
    │
    ├── Seviye kontrolü: Çalışanın rolü ≥ görevin min_role_level mi?
    │   ├── EVET → "Ben Alıyorum" butonu aktif
    │   ├── BİR ÜST SEVİYE → "🔒 Supervisor onayı gerekir" etiketiyle görünür
    │   └── İKİ+ ÜST → Görev havuzda bile görünmez
    │
    └── Çalışan "Ben Alıyorum" butonuna tıklar
        ├── initiative_claimed_by = user_id
        ├── initiative_claimed_at = NOW()
        ├── initiative_scores → +10 puan (initiative_claim)
        ├── Eğer requires_supervision → supervisor onayı beklenir
        └── Bildirim: Coach'a "Ahmet cam temizliğini üstlendi"
    
    ↓

TAMAMLAMA
    │
    ├── Zamanında → Normal tamamlama (+2 puan)
    │   └── Erken tamamlama bonusu: deadline'dan 2+ saat önce → +5 puan
    │
    ├── Gecikmeli → Gecikme kaydı (0 puan)
    │   └── Mr. Dobody: Supervisor'a "Cam temizliği henüz tamamlanmadı"
    │
    └── Yapılmadı → Gün sonu kontrolü (-5 puan)
        └── Mr. Dobody: Supervisor'a "Bugün tamamlanamayan görevler var"
    
    ↓

PERSONEL GEÇ KALMA (ayrı sistem, PDKS entegrasyonu)
    │
    ├── ≤30 dk geç → Mr. Dobody personele uyarı + eksik süre işlenir
    ├── >1 saat geç → Doğrudan supervisor'a bildirim (anlık)
    ├── Ayda 2. tekrar → Supervisor'a "tekrarlayan" uyarı
    ├── Ayda 4+ tekrar → Supervisor + Yatırımcı + CGO + Trainer'a
    └── Şube geneli sorun → CGO'ya (bireysel değil, sistemik)
    
    ↓

RAPORLAMA
    ├── Haftalık: Şube bazlı tamamlama oranı
    ├── Aylık: Çalışan bazlı inisiyatif skoru sıralaması
    └── Dashboard: Açık görev sayısı, tamamlama trendi
```

### 3.3 İnisiyatif Skor Sistemi

Fabrikadaki 5 bileşenli skor sisteminden esinlenerek, şube çalışanları için **görev performans skoru**:

| Bileşen | Ağırlık | Açıklama |
|---------|---------|----------|
| Görev tamamlama | %40 | Atanan görevlerin tamamlanma oranı |
| İnisiyatif | %25 | Açık havuzdan alınan görev sayısı |
| Zamanında tamamlama | %20 | Deadline'a uyum oranı |
| Kalite | %15 | Tamamlama kalitesi değerlendirmesi |

**İnisiyatif puan tablosu:**

| Aksiyon | Puan | Koşul |
|---------|------|-------|
| Açık görev alma (claim) | +10 | is_open_pool task'ı sahiplenme |
| Erken tamamlama | +5 | Deadline'dan 2+ saat önce |
| Zamanında tamamlama | +2 | Deadline'a kadar |
| Gecikmeli tamamlama | 0 | Deadline sonrası |
| Hiç tamamlamama | -5 | 24 saat geçtikten sonra |
| Kalite bonusu | +3 | completion_quality = 'excellent' |
| Zor görev bonusu | +5 | min_level = 'senior' görev tamamlama (bar_buddy+) |

**Hesaplama:** Haftalık otomatik (Pazartesi 04:00, fabrika skorundan sonra)

### 4.2 Yetkilendirme Kuralı

```
Coach/Trainer → Tüm şubelerin görev takvimini yönetir (global scope)
Müdür → Sadece kendi şubesinin takvimini yönetir (branch scope)  
Supervisor → Görev atar, açık havuz oluşturur, tüm seviyeleri alabilir (branch scope)
Barista → Seviye 1-3 görevleri alabilir (self scope)
Bar Buddy → Seviye 1-2 alabilir, 3 gözetimli alabilir (self scope)
Stajyer → Sadece seviye 1 alabilir (self scope)
```

**Açık havuz filtreleme mantığı:**
```
Çalışan açık havuzu açtığında:
  → Kendi rol seviyesine uygun görevleri görür
  → Bir üst seviye görevleri "🔒 Supervisor onayı gerekir" etiketi ile görür
  → İki+ üst seviye görevleri HİÇ görmez
```

## 5.3 MR. DOBODY ESCALATION FELSEFESİ — PATERN-BAZLI BİLDİRİM

### Temel Prensip: Bildirim Boğması OLMAYACAK

Mr. Dobody, Coach ve Trainer'ı her personel ihlali, her yapılmayan görev ile boğMAZ. Bunun yerine:

1. **Bireysel olaylar şube içinde kalır** (supervisor seviyesi)
2. **Mr. Dobody PATERN tespit eder** — tekrarlayan, sistemik sorunlar
3. **Sadece patern tespitinde Coach/Trainer'a bildirim gider**
4. **Coach/Trainer günlük aksiyon alabilecek kadar bildirim alır** — ne fazla ne eksik

### Escalation Zinciri (CGO'dan Önce Coach/Trainer)

```
OLAY (devamsızlık, uygunsuzluk, görev ihmali)
    │
    ├── Bireysel, tek seferlik → Şube içinde kalır (supervisor)
    │
    └── PATERN oluştu → Mr. Dobody tespit eder
            │
            ↓
        COACH / TRAINER bilgilendirilir (CGO DEĞİL!)
            │
            ├── Mr. Dobody yardımıyla şube uyarılır
            ├── Geri bildirim istenir (aydınlatma talebi)
            ├── Coach/Trainer aksiyon alır
            │   └── Bu aksiyonlar Coach/Trainer skoruna OLUMLU işlenir (+)
            │
            ↓
        CGO'YA RAPOR (aylık — sadece sonuçlar)
            │
            └── Coach/Trainer'ın aldığı aksiyonlar + sonuçları
                Şube skorları, trend, karşılaştırma
            │
            ↓
        CEO'YA ÖZET (aylık/çeyreklik — sadece büyük resim)
```

### Patern Tespit Kuralları (Mr. Dobody Zekası)

Mr. Dobody günlük olayları izler ama Coach/Trainer'a sadece patern gördüğünde bildirim gönderir:

| Patern | Tetikleme Koşulu | Bildirim Metni | Hedef |
|--------|-----------------|----------------|-------|
| Tekrarlayan devamsızlık | Aynı kişi 1 ayda 3+ kez | "Fatma H. bu ay 3. kez geç — patern oluşuyor" | Coach/Trainer |
| Şube geneli gecikme | Şubede %30+ personel aynı hafta geç | "Işıklar'da 5/12 personel bu hafta geç kaldı" | Coach/Trainer |
| Görev tamamlama düşüşü | Şube tamamlama 2 hafta üst üste %70 altı | "Lara 2 haftadır görev tamamlama düşüşünde" | Coach/Trainer |
| Checklist uyumsuzluğu | 3+ gün üst üste checklist eksik | "Markantalya 3 gündür açılış checklist'i yapılmıyor" | Coach/Trainer |
| Mesai düzeni sorunu | Şubede sistematik erken çıkış/geç giriş trendi | "Mallof'ta mesai düzeni sorunu — trend ↓" | Coach/Trainer |
| İnisiyatif düşüklüğü | Şubede 2 hafta üst üste 0 inisiyatif | "Gaziantep'te kimse inisiyatif almıyor" | Coach/Trainer |
| Performans düşüşü | Kişinin skoru 2 hafta üst üste %15+ düşüş | "{personel} performansı düşüyor — müdahale gerekebilir" | Coach/Trainer |

**Bildirim sıklığı:** Coach/Trainer'a günde maksimum 3-5 aksiyon alınabilir bildirim. Daha fazla varsa Mr. Dobody önceliklendirip en önemlileri gösterir.

### Aydınlatma Talep Akışı (Mr. Dobody Destekli İletişim)

Coach/Trainer bir patern gördüğünde şube yetkilisinden (müdür/supervisor) açıklama isteyebilir. Mr. Dobody bu süreci kolaylaştırır:

```
1. PATERN TESPİTİ
   Mr. Dobody → Coach dashboard: "Işıklar'da 3 personel tekrarlayan gecikme"
   Kart: [Aydınlatma İste] [Şubeyi Ziyaret Et] [Detay Gör]
       ↓
2. AYDINLATMA TALEBİ OLUŞTURMA
   Coach "Aydınlatma İste" tıklar
   → Mr. Dobody (yapay zeka) otomatik metin yazar:
     ┌─────────────────────────────────────────────────┐
     │ Taslak (AI tarafından oluşturuldu):             │
     │                                                  │
     │ "Sayın Basri Bey,                               │
     │  Işıklar şubesinde Mart ayında Fatma H.,        │
     │  Mehmet K. ve Ali Y.'nin tekrarlayan             │
     │  gecikmeleri tespit edilmiştir. Konunun          │
     │  aydınlatılması ve alınacak önlemler hakkında    │
     │  bilgi verilmesini rica ederim.                  │
     │  Yanıt süresi: 48 saat."                        │
     │                                                  │
     │  [Düzenle]  [Onayla ve Gönder]  [İptal]        │
     └─────────────────────────────────────────────────┘
   → Coach metni inceler, gerekirse düzenler, onaylar
   → Şube yetkilisine gönderilir (bildirim + sistemde görünür)
       ↓
3. CEVAPLAMA SÜRESİ
   Şube yetkilisi belli süre içinde (varsayılan 48 saat) yanıt yazar
   → Yanıt Coach/Trainer'a iletilir
       ↓
   ⏰ Zamanında cevaplanMAZSA:
   → Mr. Dobody: Coach'a "Işıklar aydınlatma talebine 48 saattir yanıt yok"
   → Şube yetkilisine tekrar bildirim ("Lütfen yanıtlayınız")
   → Coach bu aşamada telefon ile arayarak bilgi alabilir
       ↓
4. DEĞERLENDİRME VE AKSİYON
   Coach/Trainer yanıtı okur ve karar verir:
   ├── ✅ Yeterli açıklama → Dosya kapatılır, not düşülür
   ├── ❓ Ek bilgi gerekli → Yeni aydınlatma talebi gönderilir
   ├── ⚠️ Yazılı ikaz → İK/tutanak sistemi tetiklenir (otomatik)
   ├── 📝 Yazılı savunma istenir → Personele savunma talebi gider
   └── 📋 Tutanak düzenlenir → Disiplin kaydı oluşturulur
       ↓
5. SKOR ETKİSİ
   ├── Şube skoru: uygunsuzluk OLUMSUZ etki (−)
   ├── Personel skoru: devamsızlık/gecikme OLUMSUZ etki (−)
   └── Coach/Trainer skoru: aksiyon almış olmak OLUMLU etki (+)
       (Patern tespit → aydınlatma → çözüm = Coach performansına katkı)
```

### Aydınlatma Talebi Veri Modeli

```sql
CREATE TABLE IF NOT EXISTS clarification_requests (
  id SERIAL PRIMARY KEY,
  pattern_type VARCHAR(50) NOT NULL,     -- 'attendance', 'task_completion', 'checklist', 'schedule'
  branch_id INTEGER NOT NULL REFERENCES branches(id),
  target_user_id VARCHAR(36) REFERENCES users(id),  -- Şube yetkilisi (müdür/supervisor)
  requested_by VARCHAR(36) NOT NULL REFERENCES users(id),  -- Coach/Trainer
  ai_draft_text TEXT NOT NULL,           -- Mr. Dobody'nin yazdığı taslak metin
  final_text TEXT,                       -- Coach'un onayladığı son metin
  response_text TEXT,                    -- Şube yetkilisinin yanıtı
  response_deadline TIMESTAMP NOT NULL,  -- Yanıt son tarihi (varsayılan +48 saat)
  responded_at TIMESTAMP,
  status VARCHAR(20) DEFAULT 'draft',    -- draft, sent, responded, overdue, escalated, closed
  resolution VARCHAR(30),               -- sufficient, needs_more_info, warning_issued, defense_requested, tutanak
  related_user_ids TEXT,                -- İlgili personel ID'leri (JSON array)
  evidence_data JSONB,                  -- Patern verileri (gecikme kayıtları, görev tamamlama vb.)
  coach_action_score INTEGER DEFAULT 0, -- Coach'un bu talepten aldığı skor (+3 gönderme, +5 çözüm)
  created_at TIMESTAMP DEFAULT NOW(),
  closed_at TIMESTAMP,
  closed_by VARCHAR(36) REFERENCES users(id)
);
```

### Coach/Trainer Aksiyon Skoru

Coach veya Trainer'ın kendi performans skorunda "yönetim etkinliği" bileşeni:

| Aksiyon | Coach/Trainer Skoru Etkisi |
|---------|---------------------------|
| Patern tespitine 24 saat içinde aksiyon | +5 |
| Aydınlatma talebi gönderme | +3 |
| Yanıtı değerlendirip çözüm üretme | +5 |
| Yazılı ikaz/tutanak düzenleme (gerektiğinde) | +3 |
| Şube ziyareti sonrası denetim raporu yazma | +8 |
| Patern tespitini 72 saat görmezden gelme | -5 |
| Aydınlatma talebine yanıt gelmedi ama takip etmedi | -3 |

Bu skorlar, Coach/Trainer'ın sadece "izleyici" değil "aktif yönetici" olmasını teşvik eder.

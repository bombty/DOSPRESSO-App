BATCH-3: SOFT DELETE + ÇÖP KUTUSU (MVP)

Amaç:
- Hard delete’i kaldır.
- Kritik tablolarda deleted_at ile soft delete uygula.
- Admin panelde “Çöp Kutusu” ekranı: silinenleri listele, geri yükle, kalıcı sil.
- Silme/geri yükleme işlemleri Audit Log’a düşsün.
- Mevcut sistemi kırmadan minimum değişiklikle ilerle.

Kapsam (Faz-1 / MVP tablolar):
- users
- branches
- tasks
- equipment
- checklists
- academy_recipes (veya recipes tablosu sistemde hangi isimdeyse)
- training_modules (veya training_* tabloları sistemde hangi isimdeyse)
- shifts

Yapılacaklar (sırayla):

1) DB Şema
- Yukarıdaki tablolara nullable `deleted_at timestamp` alanı ekle.
- İndeks: (deleted_at), ayrıca sık filtrelenenlerde (branch_id, deleted_at) gibi compound index ekle.

2) API Davranışı
- Var olan DELETE endpoint’lerini hard delete yerine soft delete’e çevir:
  UPDATE ... SET deleted_at = NOW()
- Tüm LIST/GET endpoint’lerinde varsayılan filtre:
  WHERE deleted_at IS NULL
- Admin yetkisi olanlar için opsiyon: includeDeleted=true gelirse filtreyi kaldırabilsin.

3) Çöp Kutusu API (admin)
- GET  /api/admin/trash?table=users&limit=50&offset=0&search=...  (table paramı zorunlu)
- PATCH /api/admin/trash/:table/:id/restore  -> deleted_at=NULL
- DELETE /api/admin/trash/:table/:id/permanent -> gerçek hard delete (sadece admin)
- Güvenlik: table allowlist kullan. Kullanıcının istediği herhangi bir tabloya erişim olmasın.

4) UI (Admin)
- Admin / Sistem altında “Çöp Kutusu” sekmesi ekle.
- Tablo seçici (Users/Branches/Tasks/Equipment/Checklists/Recipes/Training/Shifts).
- Liste: ad, id, branch adı, silinme zamanı, silen kişi (audit’ten bağlanabiliyorsa), Aksiyonlar: Geri Yükle / Kalıcı Sil.
- Kalıcı sil için ekstra onay modalı: “DELETE” yazdır.

5) Audit Log Entegrasyonu
- soft delete: event_type="soft_delete", resource, resourceId, scopeBranchId, actorRole, requestId
- restore: event_type="restore"
- permanent delete: event_type="hard_delete"
- details içine before/after koy (en azından deleted_at değişimi)

6) Test Senaryosu (kendin doğrula)
- 1 kullanıcıyı sil → sistemden kaybolmalı
- Çöp kutusunda görünmeli
- Geri yükle → tekrar görünmeli
- Audit log’da 3 kayıt oluşmalı (soft_delete/restore)
- CSV export bu kayıtları indirebilmeli

Kısıtlar:
- Mevcut tabloların isimleri farklıysa önce koddan gerçek tablo isimlerini bul ve kapsamı doğru tablo isimleriyle uygula.
- Cascade delete’leri bu MVP’de kaldırma; çünkü soft delete ile hard cascade tetiklenmeyecek. Hard delete sadece “permanent” endpoint’inde olacak.
- Her batch sonunda kısa özet yaz ve benden onay istemeden Batch-4’e geçme.
